# פרוטוקול תקלות — קול נובה

## רמות תקלה

| רמה | תיאור | זמן תגובה |
|-----|--------|-----------|
| 🔴 **קריטי** | ייעוץ רפואי ניתן / מחיר שגוי / מידע רפואי נחשף | מיידי (< 15 דקות) |
| 🟠 **גבוה** | תור נקבע בשעה שגויה / שם מטופל שגוי / שיחה ניתקה | < 1 שעה |
| 🟡 **בינוני** | Agent לא הבין בקשה / שפה לא נתמכת / תגובה איטית | < 4 שעות |
| 🟢 **נמוך** | שיפור prompt / שינוי ניסוח / בקשת פיצ'ר | < 24 שעות |

## תהליך טיפול

### שלב 1 — זיהוי והתראה (מיידי)
- בדוק את לוגי Vapi של השיחה
- הודע ללקוח מיד ב-WhatsApp:
```
שלום [שם], שמתי לב לבעיה בשיחה מ-[שעה].
אני מטפל בזה עכשיו ומעדכן אותך תוך שעה.
```

### שלב 2 — תיקון (< 1 שעה)
- עדכן את ה-system prompt ב-Vapi
- אם הבעיה מחייבת — הוסף כלל מפורש ב-prompt
- בדוק שיחה דומה (dry run) לוודא שהתיקון עובד

### שלב 3 — בדיקת לוגים
- עבור על 10 השיחות האחרונות
- חפש דפוס דומה (האם הבעיה חד-פעמית או מערכתית?)
- תעד ב-incident log

### שלב 4 — Fallback אנושי
אם הבעיה מערכתית ולא ניתנת לתיקון מיידי:
- הפעל העברת שיחות ישירה (forward) לנייד המרפאה
- הודע ללקוח: "המערכת מועברת למצב ידני עד שנתקן"
- אל תשאיר agent פגום פעיל

### שלב 5 — סקירה (< 24 שעות)
- סקירה מלאה עם הלקוח
- הסבר מה קרה ומה תוקן
- עדכן את מסמך ה-onboarding אם צריך
- תעד ב-incident log לשימוש עתידי

## כללים

1. **שקיפות מלאה** — אף פעם לא להסתיר תקלה מלקוח
2. **תיקון לפני הסבר** — קודם תקן, אחרי זה תסביר
3. **מניעה > תיקון** — אחרי כל תקלה, הוסף guard rail ב-prompt
4. **Fallback תמיד מוכן** — העברה לנייד צריכה להיות מוגדרת מראש
5. **אפס סבלנות לשגיאות רפואיות** — ייעוץ רפואי = כיבוי מיידי + בדיקה מלאה
